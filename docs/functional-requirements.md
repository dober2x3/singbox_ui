# Функциональные требования — Sing-box UI

## 1. Назначение системы

**Sing-box UI** — это веб-интерфейс для управления sing-box (прокси/VPN-решение). Система работает по принципу Docker-in-Docker: веб-интерфейс (Go backend + Next.js frontend) запускается в контейнере и управляет отдельным контейнером sing-box через Docker socket. Пользователь настраивает конфигурацию через браузер, а бэкенд применяет её к работающему контейнеру sing-box.

## 2. Функциональные блоки

### 2.1 Управление контейнером sing-box (Container Lifecycle)

| ID | Требование | Описание |
|:---|:-----------|:---------|
| **CL-01** | Запуск контейнера | Пользователь может запустить контейнер sing-box с текущей конфигурацией через Docker API |
| **CL-02** | Остановка контейнера | Пользователь может остановить работающий контейнер sing-box |
| **CL-03** | Перезапуск контейнера | Пользователь может перезапустить контейнер с обновлённой конфигурацией |
| **CL-04** | Мониторинг статуса | Система отображает статус контейнера (запущен/остановлен) |
| **CL-05** | Валидация конфигурации | Перед запуском контейнера система валидирует конфигурацию, запуская временный контейнер с флагом `check` |
| **CL-06** | Версионирование образа | Система отслеживает версию sing-box образа и поддерживает обновление |
| **CL-07** | Логи контейнера | Система стримит логи работающего контейнера в реальном времени |

**Реализация**: `server/services/docker.go` — DockerService с методами CreateContainer, StartContainer, StopContainer, RemoveContainer, StreamLogs, CheckConfig. Оркестрация: `server/services/singbox.go`.

### 2.2 Редактор конфигурации (Configuration Management)

| ID | Требование | Описание |
|:---|:-----------|:---------|
| **CM-01** | Управление входящими подключениями (Inbounds) | Пользователь может создавать/редактировать/удалять inbound-протоколы: VLESS, VMess, Trojan, Shadowsocks, Hysteria2, TUIC, Naive, Mixed, ShadowTLS, AnyTLS, HTTP, WireGuard |
| **CM-02** | Управление исходящими подключениями (Outbounds) | Пользователь может настраивать proxy outbound, URL-test (балансировщик), direct, block, dns-out |
| **CM-03** | DNS-конфигурация | Пользователь может настраивать DNS-серверы (UDP, TCP, HTTPS, TLS, hosts), правила маршрутизации DNS с поддержкой rule_set, глобальные настройки (final server, independent cache) |
| **CM-04** | Правила маршрутизации (Routing) | Пользователь может создавать правила маршрутизации трафика на основе geoip, geosite, port, protocol, network и т.д. |
| **CM-05** | Шаблоны DNS | Система предоставляет готовые шаблоны DNS-конфигурации для распространённых сценариев |
| **CM-06** | Сохранение и применение | Изменения конфигурации сохраняются в JSON-файл на хост-системе и применяются к контейнеру |
| **CM-07** | Слияние конфигураций | Frontend поддерживает частичное обновление конфигурации через механизм слияния (merge-config) |
| **CM-08** | Графический интерфейс форм | Каждый протокол имеет специализированную форму с полями для его уникальных параметров (транспорт, TLS, аутентификация) |
| **CM-09** | WireGuard VPN | Управление пирами WireGuard, генерация клиентской конфигурации с QR-кодом и скачиванием файла |

**Реализация**: `frontend/components/inbound/*` (12 форм протоколов + barrel index), `frontend/components/outbound/`, `frontend/components/dns/`, `frontend/lib/store/singbox-config.ts` (Zustand store), `server/services/singbox_config.go`.

### 2.3 Управление подписками (Subscription Management)

| ID | Требование | Описание |
|:---|:-----------|:---------|
| **SM-01** | Добавление подписки | Пользователь может добавить URL прокси-подписки |
| **SM-02** | Парсинг подписок | Система автоматически парсит подписки в форматах: VMess-URL, VLESS-URL, Trojan-URL, Shadowsocks-URL, Clash YAML |
| **SM-03** | Обновление подписок | Пользователь может принудительно обновить/освежить список узлов из подписки |
| **SM-04** | Удаление подписки | Пользователь может удалить подписку и её узлы |
| **SM-05** | Отображение списка узлов | Система отображает все распарсенные прокси-узлы с их параметрами |
| **SM-06** | Автоматическое обновление | Система поддерживает фоновое автоматическое обновление подписок по расписанию |

**Реализация**: `server/services/subscription.go`, `server/services/subscription_service.go`, `server/handlers/subscription.go`, `frontend/components/subscription-manager.tsx`, `frontend/lib/store/subscriptions.ts`.

### 2.4 Мониторинг здоровья узлов (Node Health Probing)

| ID | Требование | Описание |
|:---|:-----------|:---------|
| **NP-01** | Проверка TCP-доступности | Система выполняет TCP-проверку (handshake) на указанные прокси-узлы |
| **NP-02** | Проверка HTTP-доступности | Система выполняет HTTP-запрос к узлам с замером времени ответа |
| **NP-03** | Кольцевой буфер истории | Система хранит историю результатов проверок в кольцевом буфере для расчёта процента успешных проверок |
| **NP-04** | Конкурентные проверки | Проверки выполняются конкурентно с ограничением через семафор (канал с буфером) |
| **NP-05** | CRUD проверок | Пользователь может создавать, просматривать, обновлять и удалять конфигурации проверок (цель, интервал, тип) |
| **NP-06** | Запуск/остановка | Пользователь может запускать и останавливать мониторинг для отдельных узлов |

**Реализация**: `server/services/prober.go` (ядро движка проверок), `server/services/prober_manager.go`, `server/services/prober_service.go`, `frontend/components/prober/*`.

### 2.5 Тестирование скорости (Speed Testing)

| ID | Требование | Описание |
|:---|:-----------|:---------|

Группы маршрутов Gin:

| Метод | Путь | Описание |
|:------|:-----|:---------|

## 5. Пользовательские сценарии (User Flows)

### 5.1 «Быстрый старт»
1. Пользователь запускает Docker-контейнер Sing-box UI
2. Открывает веб-интерфейс
3. Система автоматически генерирует дефолтную конфигурацию sing-box
4. Пользователь добавляет inbound-протокол (например, VLESS)
5. Настраивает TLS-сертификат (или использует ACME)
6. Добавляет подписку с прокси-узлами
7. Запускает контейнер sing-box → трафик пошёл

### 5.2 «Мониторинг и отладка»
1. Пользователь открывает логи контейнера в реальном времени
2. Настраивает health probe на проблемный узел
3. Запускает speed test для сравнения узлов
4. Анализирует результаты и переключается на лучший узел

### 5.3 «Обновление конфигурации»
1. Пользователь редактирует DNS-правила или routing-правила
2. Система валидирует конфигурацию через временный Docker-контейнер
3. При успехе — конфигурация сохраняется и применяется
4. Если валидация не пройдена — пользователь получает ошибку с описанием
