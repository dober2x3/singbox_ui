# API Documentation Design (Swagger/OpenAPI)

**Date:** 2026-06-01
**Author:** AI Agent
**Status:** Draft

## 1. Executive Summary

Add automatic OpenAPI (Swagger) documentation to the Go backend using **swaggo** — the de facto standard for Go + Gin API documentation. All ~57 endpoints will be fully documented with request parameters, response schemas, error codes, and examples. The documentation will be served via Swagger UI at `/swagger/index.html`.

## 2. Current State

```
server/
├── main.go                   # Entry point, Gin router, route grouping
├── internal/
│   ├── certificate/          # 5 endpoints
│   ├── prober/               # 12 endpoints
│   ├── singbox/              # 15 endpoints
│   ├── speedtest/            # 3 endpoints
│   ├── subscription/         # 7 endpoints
│   ├── warp/                 # 5 endpoints
│   └── wireguard/            # 8 endpoints
```

**Problems:**
- Zero API documentation — consumers (frontend, CLI, other services) must read Go source code
- Request/response structures are defined inline as anonymous structs inside handler methods — invisible to tooling
- No shared error types — each handler returns ad-hoc error formats
- No examples or descriptions for fields

## 3. Target Architecture

```
server/
├── main.go                   # + swagger route, + import docs
├── internal/
│   └── docs/                 # New package
│       ├── docs.go           # Root swaggo annotations (title, version, tags)
│       └── errors.go         # ErrorResponse shared type
├── internal/certificate/
│   ├── handler.go            # + swaggo annotations
│   ├── models.go             # Named request/response types
│   └── ...rest unchanged
├── ...other domains/         # Same pattern
├── docs/                     # Generated (gitignored or committed — TBD)
│   ├── swagger.json
│   ├── swagger.yaml
│   └── docs.go               # Auto-generated by swag init
```

## 4. Annotation Patterns

### 4.1 Root metadata (`internal/docs/docs.go`)

```go
package docs

// @title           Singbox UI API
// @version         1.0
// @description     API for managing sing-box, WireGuard, subscriptions, probing, and WARP
// @host            localhost:8080
// @BasePath        /api
// @schemes         http
//
// @tag.name        subscription
// @tag.description Subscription management
// @tag.name        singbox
// @tag.description sing-box container lifecycle
// @tag.name        wireguard
// @tag.description WireGuard utilities
// @tag.name        certificate
// @tag.description Self-signed certificates
// @tag.name        reality
// @tag.description REALITY keypair & TLS checks
// @tag.name        prober
// @tag.description Node probing
// @tag.name        speedtest
// @tag.description Speed tests
// @tag.name        warp
// @tag.description WARP registration & scanning
// @tag.name        system
// @tag.description System endpoints (health, etc.)
```

### 4.2 Standard error type (`internal/docs/errors.go`)

```go
package docs

type ErrorResponse struct {
	Error string `json:"error" example:"invalid subscription URL"`
}
```

### 4.3 Handler annotation example

```go
// GetSubscriptions returns a list of all subscriptions
// @Summary      Get all subscriptions
// @Description  Returns all subscriptions with total node counts
// @Tags         subscription
// @Success      200  {array}   SubscriptionResponse
// @Failure      500  {object}  docs.ErrorResponse
// @Router       /subscription [get]
func (h *Handler) GetSubscriptions(c *gin.Context) { ... }
```

### 4.4 Request/response types — from inline to named

**Before (current pattern):**
```go
func (h *Handler) AddSubscription(c *gin.Context) {
    var req struct {
        Name      string `json:"name"`
        URL       string `json:"url"`
        UserAgent string `json:"user_agent"`
    }
```

**After (types in models.go):**
```go
// AddSubscriptionRequest request body for adding a subscription
type AddSubscriptionRequest struct {
	Name      string `json:"name" example:"My VPN Server"`
	URL       string `json:"url" example:"https://example.com/sub"`
	UserAgent string `json:"user_agent" example:"clash-meta" enums:"clash-meta,sing-box,v2ray"`
}

// AddSubscriptionResponse response after adding a subscription
type AddSubscriptionResponse struct {
	ID        string `json:"id" example:"sub_abc123"`
	Name      string `json:"name" example:"My VPN Server"`
	NodeCount int    `json:"node_count" example:"12"`
}
```

## 5. Route Registration

In `main.go`:

```go
import (
	swaggerFiles "github.com/swaggo/files"
	ginSwagger "github.com/swaggo/gin-swagger"
	_ "github.com/dober2x3/singbox_ui/server/internal/docs"
)

// In setupRoutes or main():
r.GET("/swagger/*any", ginSwagger.WrapHandler(swaggerFiles.Handler))
```

## 6. Implementation Plan (Phased)

### Phase 0: Foundation
**Files:** `go.mod`, `internal/docs/`, `main.go`
- Install swaggo CLI
- Add go.mod dependencies
- Create `internal/docs/docs.go` with root annotations
- Create `internal/docs/errors.go` with `ErrorResponse`
- Add swagger route to `main.go`
- Run `swag init -g main.go` to verify generation works

### Phase 1: certificate + reality (5 endpoints)
**Files:** `internal/certificate/handler.go`, `internal/certificate/models.go`
- Create `models.go` with named request/response types
- Add swaggo annotations to all 5 handlers
- Generate docs, verify in Swagger UI

### Phase 2: speedtest (3 endpoints)
**Files:** `internal/speedtest/handler.go`, `internal/speedtest/models.go`
- Simple domain, good for validating the pattern

### Phase 3: wireguard (8 endpoints)
**Files:** `internal/wireguard/handler.go`, `internal/wireguard/models.go`

### Phase 4: warp (5 endpoints)
**Files:** `internal/warp/handler.go`, `internal/warp/models.go`

### Phase 5: subscription (7 endpoints)
**Files:** `internal/subscription/handler.go`, `internal/subscription/models.go`
- Note: `subscription/models.go` already exists — needs swaggo tags + examples

### Phase 6: prober (12 endpoints)
**Files:** `internal/prober/handler.go`, `internal/prober/models.go`

### Phase 7: singbox (15 endpoints)
**Files:** `internal/singbox/handler.go`, `internal/singbox/models.go`
- Largest domain, saved for last when the pattern is well-established

### Phase 8: system + main.go (2 endpoints)
- Health check, reality keypair (inline handlers in main.go)

## 7. Testing & Verification

- After each phase: `cd server && swag init -g main.go` must succeed with no errors
- After each phase: start the server and verify `/swagger/index.html` renders the new endpoints
- All existing tests must pass: `go test ./...`
- Lint must pass: `golangci-lint run ./...`

## 9. Out of Scope

- API versioning (no `/v1/` prefix — not currently used)
- Authentication/authorization (not currently in the backend)
- Client SDK generation (can be added later from the generated spec)
- Documentation of the frontend API interaction layer
